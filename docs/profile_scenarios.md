# 프로필 화면 시나리오

## 관련 페이지

| 경로 | 역할 |
|------|------|
| `/join` | 회원가입 / 로그인 |
| `/profile` | 프로필 추가 / 삭제 |
| `/my` | 계정 정보 확인 · 로그아웃 · 탈퇴 |

로그인 상태는 `localStorage["kkachi_member_id"]`로 판단한다.

---
## 시나리오 1 — 신규 가입

2단계 플로우로 진행된다. 모든 신규 계정은 자기 자신의 프로필을 반드시 갖는다.

**1단계 — 계정**
1. `/join` 진입 → 이름 + 이메일 입력
2. `POST /members` 호출
   - 새 이메일 → 멤버 생성 후 반환
   - **기존 이메일** → 기존 멤버 그대로 반환 (비밀번호 없음, 이메일이 곧 식별자)
3. `localStorage["kkachi_member_id"]` 저장
4. `GET /members/{id}/profiles` 호출
   - 프로필 0개 → 2단계로 전환
   - 프로필 1개 이상 → `/` redirect (재방문자)

**2단계 — 내 사주 등록**
1. 이름(1단계에서 pre-fill, 수정 가능) · 생년월일 · 태어난 시간 · 도시 · 성별 입력
2. `POST /members/{id}/profiles` 호출
3. 성공 시 `/` redirect

---

## 시나리오 2 — 재방문 (자동 로그인)

- `localStorage["kkachi_member_id"]` 있으면 이미 로그인 상태
- `/join` 재진입 시 별도 리다이렉트 없음 (폼 표시)
  - 단, 계정 확인 후 프로필이 이미 있으면 1단계에서 바로 `/` redirect
- `/profile`, `/my` 진입 시 localStorage 없으면 `/join` redirect

---

## 시나리오 3 — 프로필 추가

1. `/profile` → "프로필 추가" 클릭 → `ProfileForm` 표시
   - 프로필 수가 10개이면 버튼 숨김 ("최대 10개" 안내 표시)
2. 입력 항목: 이름(필수) · 생년월일 · 태어난 시간(시주, "모르겠어요" 선택 시 12:00 처리) · 도시 · 성별
3. 도시 기본값: `ipapi.co` IP 위치 감지 → 자동 주입 (미감지 시 "Seoul")
4. `POST /members/{id}/profiles` 호출
   - 백엔드에서 10개 초과 시 400 반환
5. 성공 시 목록 맨 앞에 추가, 폼 닫힘

### 엣지 케이스
- 시간 모를 때: `birth_dt`의 시간부 = `12:00:00` (정오 처리)
- 도시 감지 실패: 기본값 "Seoul" 유지, 사용자가 직접 수정 가능
- API 실패: "프로필 추가에 실패했습니다." 인라인 에러, 폼 유지

---

## 시나리오 4 — 프로필 수정 (자기 자신 프로필)

자기 자신 프로필(`is_self=true`)은 삭제 불가 — 수정만 가능하다.

1. 프로필 카드에 "수정" 버튼 표시 (삭제 버튼 없음)
2. "수정" 클릭 → 카드가 인라인 편집 폼으로 전환 (기존 값 pre-fill)
3. `PATCH /members/{id}/profiles/{profile_id}` 호출
4. 성공 시 카드 업데이트, 편집 폼 닫힘

## 시나리오 5 — 프로필 삭제 (타인 프로필)

`is_self=false` 프로필만 삭제 가능하다.

1. 프로필 카드 "삭제" 클릭 → 카드 내 확인 버튼으로 전환 (2단계)
2. "삭제 확인" 클릭 → `DELETE /members/{id}/profiles/{profile_id}` 호출
   - 백엔드에서 `is_self=true` 프로필 삭제 시도 시 409 반환
3. 성공 시 목록에서 즉시 제거
4. API 실패 시 버튼 원상복구 (에러 메시지 없음 — 현재 미구현)

### 주의
- 삭제된 프로필에 연결된 `analyses`, `daily_fortunes`, `compatibilities`는 DB cascade 삭제

---

## 시나리오 6 — 로그아웃

1. `/my` → 헤더 우측 "로그아웃" 클릭
2. `localStorage["kkachi_member_id"]` 제거
3. `/` redirect

---

## 시나리오 7 — 회원 탈퇴

1. `/my` → "탈퇴하기" 클릭 → 이메일 입력 폼 표시
2. 가입 이메일과 일치해야 "탈퇴 확인" 활성화
3. `DELETE /members/{id}` 호출
4. 성공 시 localStorage 제거 → `/` redirect
5. 실패 시 "탈퇴 처리 중 오류가 발생했습니다." 표시, 폼 유지

### 주의
- cascade 삭제: 멤버 → 프로필 → 분석·궁합·일진 캐시 전부 삭제됨
- 탈퇴 후 같은 이메일로 재가입하면 새 멤버로 시작 (이전 데이터 없음)

---

## 시나리오 8 — 비로그인 상태에서 보호 페이지 접근

| 접근 경로 | 처리 |
|----------|------|
| `/profile` 직접 진입 | `/join` replace redirect |
| `/my` 직접 진입 | `/join` replace redirect |
| `/my` 진입 후 API 404 (멤버 삭제됨) | localStorage 제거 → `/join` replace redirect |

---

## 미결 사항 / 개선 검토

- [ ] 프로필 상한 10개 제한 (백엔드 400 + 프론트 버튼 숨김) — 구현 완료
- [ ] 프로필 삭제 실패 시 에러 메시지 미노출
- [ ] `/join` 진입 시 이미 로그인 상태면 `/profile` redirect 고려
- [ ] 프로필 상한 제한 없음 (무제한 추가 가능)
- [ ] 시주 "모르겠어요" = 정오(12:00) 처리 — 사용자에게 안내 문구 필요 여부
